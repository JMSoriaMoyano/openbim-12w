# DT-S5L-01 · Resolución

**Deuda técnica:** Aparente discrepancia entre schema reportado por `ifcopenshell` (`IFC4`) y cabecera STEP de `AC20-FZK-Haus.ifc` (sospecha de `IFC2X3`).
**Identificada en:** S5·L Bloque B (Lun 08/06/2026).
**Resuelta en:** S5·X Bloque A (Mié 10/06/2026).
**Estado:** Cerrada como **falso positivo**.

---

## 1. Hipótesis inicial (S5·L)

Durante la lectura geométrica del modelo en S5·L, al cruzar el `model.schema` reportado por `ifcopenshell` con la cabecera del fichero IFC, se sospechó que la cabecera STEP declaraba `IFC2X3` mientras `ifcopenshell` lo cargaba como `IFC4`, lo que habría implicado un conflicto de versión de schema.

## 2. Investigación (S5·X Bloque A)

### 2.1 Cabecera STEP cruda (lectura sin parser IFC)

```
1| ISO-10303-21;
2| HEADER;FILE_DESCRIPTION(('ViewDefinition [, QuantityTakeOffAddOnView, ...']),'2;1');
3| FILE_NAME('S:\\[IFC]\\[COMPLETE-BUILDINGS]\\FZK-MODELS\\FZK-Haus\\ArchiCAD-20\\AC20-FZK-Haus.ifc',
           '2016-12-21T17:54:06',('Architect'),('Building Designer Office'),
           'The EXPRESS Data Manager Version 5.02.0100.09 : 26 Sep 2013',
           'IFC file generated by GRAPHISOFT ARCHICAD-64 20.0.0 GER FULL Windows version (IFC2x3 add-on version: 4009 GER FULL).',
           'The authorising person');
4| FILE_SCHEMA(('IFC4'));      ←  ← ← DECLARACIÓN OFICIAL DEL SCHEMA
5| ENDSEC;
```

### 2.2 Lo que reporta `ifcopenshell`

| Campo | Valor |
|---|---|
| `model.schema` | `'IFC4'` |
| `model.schema_identifier` | `'IFC4'` |
| `model.header.file_schema.schema_identifiers` | `('IFC4',)` |
| `model.header.file_name.originating_system` | `'IFC file generated by GRAPHISOFT ARCHICAD-64 20.0.0 GER FULL Windows version (IFC2x3 add-on version: 4009 GER FULL).'` |

### 2.3 Evidencias estructurales adicionales

- `IfcDoor` (#17468) tiene atributo `PredefinedType` — atributo introducido en IFC4, **no existe** en IFC2X3.
- Coexisten `IfcWall` (13) e `IfcWallStandardCase` (13) — comportamiento típico IFC4 (en IFC2X3 lo habitual es solo `IfcWallStandardCase`).
- `IfcProject` expone atributos modernos coherentes con IFC4.

## 3. Causa raíz de la confusión

El campo `FILE_NAME.originating_system` contiene la cadena `IFC2x3 add-on version: 4009`. Esto es el **nombre comercial del add-on de exportación IFC de ArchiCAD 20**, que históricamente se llamó "IFC2x3 add-on" y mantuvo ese nombre incluso cuando comenzó a exportar IFC4. La línea 12 de la sección DATA confirma este patrón:

```
#11= IFCAPPLICATION(#10,'20.0.0','ARCHICAD-64','IFC2x3 add-on version: 4009 GER FULL');
```

`IfcApplication.ApplicationFullName` apunta al nombre comercial del add-on, no al schema. El schema oficial es **siempre** `FILE_SCHEMA` en la cabecera STEP.

## 4. Conclusión

- No hay discrepancia real. El modelo es **IFC4** y `ifcopenshell` lo reporta correctamente.
- La auditoría EIR existente (`s4s_audit_eir.py`) opera correctamente sobre el modelo.
- El script de lectura geométrica (`s5l_ifc_geom_read.py`) opera correctamente.
- No se requiere ningún cambio de código ni de configuración.

## 5. Aprendizaje formativo (registro CDE)

**Regla operativa OpenBIM v0.1:**

> Para determinar el schema IFC de un fichero, leer **siempre** `FILE_SCHEMA(...)` en la cabecera STEP (o equivalentemente `model.schema` / `model.header.file_schema.schema_identifiers` en ifcopenshell). **Nunca** inferir el schema desde `FILE_NAME.originating_system` ni desde `IfcApplication.ApplicationFullName`, que reflejan el nombre comercial del software/add-on exportador y pueden incluir referencias a versiones de schema antiguas por motivos históricos de branding.

Esta regla se incorporará a una futura sección "Lectura crítica de cabeceras IFC" en `docs/S3L_ifc_glosario.md` o equivalente en S6·L cuando se aborde calidad de datos.
